# Calorie Tracker App – Reflection

## Overview
After defining system requirements and stakeholder concerns, the next critical step was to model how users actually interact with the system (use cases) and define comprehensive test scenarios. Creating a UML use case diagram, writing detailed specifications for 8 critical use cases, and developing 26 test cases (15 functional, 11 non-functional) transformed abstract requirements into actionable, testable flows. This work revealed implementation complexities and edge cases that pure requirement statements don't fully capture, and demonstrated the tight feedback loop between requirements, design, and testing.

## Challenges Faced

### 1. Balancing Breadth and Depth in the Use Case Model
The initial impulse was to create as many use cases as possible to show comprehensive coverage. However, I realized that 11 main use cases were sufficient to capture all 12 functional requirements and address every stakeholder concern, without creating redundant or overly granular flows. The challenge was knowing when to combine flows (e.g., Edit and Delete as alternative flows within UC-05) versus when to separate them (e.g., Create Meal Log and Search Food as inclusion relationships). The use case diagram had to balance visual clarity with precision: too few use cases and I risked missing important interactions; too many and the diagram became cluttered. I settled on the principle that each use case should represent a distinct user goal, not an implementation detail. This discipline made the diagram more valuable as a communication tool.

### 2. Translating Requirements into Concrete Pre/Postconditions and Alternative Flows
Functional requirements like "FR-01: The system shall allow users to create a meal log" are relatively abstract. Turning this into a detailed use case specification required asking: What happens before the user can start? What state changes occur? What goes wrong? Writing the preconditions (user authenticated, form accessible, database populated) and postconditions (meal saved, totals updated, history reflects entry) forced me to clarify implicit assumptions embedded in the original requirement. The alternative flows revealed that a simple-sounding feature like meal logging actually has multiple failure modes: missing food items, invalid portion sizes, connectivity issues, and potential duplicate entries. These alternative flows became the foundation for validation tests (TC-014, TC-015) and resilience scenarios. This exercise showed how requirements elicitation is iterative—the act of modeling use cases revealed gaps that needed to be addressed in requirements or tests.

### 3. Designing Test Cases that Remain Traceable and Realistic
With 12 functional requirements, it would have been easy to create one test case per requirement and call it complete. However, the challenge was making test cases realistic and comprehensive without assuming a fully built application existed. I mapped each test case to a specific requirement ID and use case, ensuring traceability (TC-001 maps to FR-01 and UC-01). A trickier challenge was writing test procedures that could be executed manually during early development or automated in continuous integration later, without being tied to implementation details. I wrote each test case to focus on inputs and expected outputs rather than technical steps like "click the button with ID 'btn-save'." This approach makes the test plan resilient to UI changes and implementation technology shifts.

### 4. Covering Non-Functional Requirements Without Oversimplifying
Non-functional requirements like "NFR-P1: The dashboard shall load within 2 seconds" need test scenarios, but testing performance requires specialized tooling (Lighthouse, k6) that may not be available during initial development. TC-100–TC-110 specify *how* to measure performance, security, usability, and maintainability without assuming specific tools. For example, TC-105 (Secrets not exposed) can be validated by code review and manual inspection, or by automated scanning tools—the test procedure remains valid either way. This approach makes the test plan practical across different development environments and team structures.

### 5. Determining the Right Level of Detail in Alternative Flows
In UC-01 (Create Meal Log), I included four alternative flows (Food Not Found, Invalid Portion, Duplicate Entry, Offline Mode). Each addresses a real concern or hints at data quality issues. However, I had to decide: How deep should each alternative go? The "Offline Mode" alternative mentions local queueing and syncing, but the current system scope doesn't include offline support. I kept it to show extensibility while acknowledging that only basic alternatives (A1, A2) must be implemented initially. This balance—being comprehensive about *possible* interactions while remaining realistic about *current* scope—proved crucial for maintaining clarity without limiting future growth.

### 6. Maintaining Consistency Between Requirements, Use Cases, and Tests
Each functional requirement (FR-01 through FR-12) needed to map to at least one use case, and each use case needed to map to at least one test case. This three-way traceability could become brittle: if a requirement changed, every dependent use case and test could break. To manage this, I created explicit traceability tables in both USE_CASES.md and TEST_AND_USE_CASES.md. I also noticed that some requirements like FR-10 (persistence) and FR-11 (environment variables) are cross-cutting—they describe system infrastructure rather than user-facing flows. These ended up in UC-08 (Configure Deployment) rather than user workflows, which felt correct from a stakeholder perspective (System Admin is a user of this flow). This decision-making process highlighted how use cases provide a reality check on requirement organization.

## Lessons Learned

1. **Use Cases Clarify Assumptions**: Writing preconditions and postconditions surfaced hidden assumptions in requirements. FR-03 "display total calories" implicitly assumes meals exist and are aggregated—but what if today is the user's first day? UC-03 addresses this with alternative flow A1 (no meals logged yet).

2. **Alternative Flows Are Requirements Too**: Initially, I treated alternative flows as optional details. I realized they define the system's robustness. A3 in UC-05 (concurrent edit conflict) doesn't currently affect a single-user app but becomes critical in multi-user deployment. Including such flows demonstrates foresight.

3. **Test Case Granularity Matters**: TC-001 (log meal with valid details) is an end-to-end test; more granular tests (TC-014, TC-015) cover specific failure modes. This combination provides good balance between coverage and maintainability.

4. **Stakeholder Concerns Drive Use Case Selection**: Initially, I tried to map requirements mechanistically to use cases. The breakthrough came when I focused on stakeholder concerns: What does the Primary User need? (UC-01, UC-02, UC-03, UC-05) What does the Nutrition Coach need? (UC-04, UC-07) What do operators need? (UC-08) This stakeholder-driven approach ensured the model answered real business questions.

## Conclusion
Building a complete use case model and test plan revealed gaps and inconsistencies in the original requirements that would only surface during implementation or user testing. The exercise of writing detailed specifications with preconditions, basic flows, and alternative flows forced clarity. The test plan provided a blueprint for validation: both automated testing during development and manual testing during deployment. The documentation now tells a coherent story from stakeholder needs → requirements → interactions → tests, making it significantly easier for future developers to understand not just *what* the system should do, but *why* each feature exists and *how* to verify it works correctly.
