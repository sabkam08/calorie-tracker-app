# Calorie Tracker App – Reflection

## Overview
This reflection presents the documentation set as an integrated requirements and delivery package. Each file contributes a distinct layer of analysis, beginning with problem definition and progressing through stakeholder analysis, requirements specification, use cases, testing, Agile planning, and GitHub Project workflow design. The principal strength of the documentation is traceability: each document supports the next and reduces ambiguity in the overall project record.

## Specification Reflection
`SPECIFICATION.md` established the project boundary by defining the problem, the target domain, and the implementation scope. Its most important contribution was the deliberate decision to focus on core calorie-tracking functionality rather than expanding into advanced features. That decision improved coherence across the documentation set and ensured that the remaining artefacts were realistic for the assignment timeframe. The specification also established a technical direction that supports rapid delivery while remaining maintainable.

## Stakeholder Analysis Reflection
`STAKEHOLDER_ANALYSIS.md` required a more disciplined interpretation of the system than a narrow end-user perspective would have allowed. The document distinguishes between primary users, professional users, technical contributors, commercial interests, and external knowledge partners. That separation improved prioritisation because it clarified that different stakeholders influence the project in different ways. The power-interest matrix and engagement strategy further strengthened the analysis by showing which groups require close management, which require regular communication, and which can be monitored with lower intensity.

## System Requirements Reflection
`SYSTEM_REQUIREMENTS.md` translated stakeholder concerns into testable functional and non-functional requirements. Its strongest characteristic is specificity. Each requirement is written so that it can later be verified, which improves clarity and reduces interpretation risk. The non-functional requirements are particularly valuable because they ensure the system is assessed not only as a set of visible features but also as a deployable, maintainable, scalable, secure, and performant application.

## Use Case Reflection
`USE_CASES.md` converted abstract requirements into structured user interactions. This stage exposed assumptions that were not visible in the requirement statements alone. Preconditions, postconditions, basic flows, and alternative flows made the behaviour of the system easier to evaluate and communicate. The relationships between actors and use cases also clarified which stakeholders are directly involved in the current scope and which are represented primarily through supporting or future-oriented concerns.

## Test and Use Case Reflection
`TEST_AND_USE_CASES.md` provided the clearest bridge between specification and verification. It demonstrates how functional requirements can be validated through specific test scenarios and how non-functional requirements can be expressed in measurable terms. This file is important because it transforms descriptive statements into observable checks. Without it, the documentation would explain what the system should do, but not how the system’s success would be confirmed.

## Agile Planning and GitHub Project Reflection
`AGILE_PLANNING.md` moved the work from analysis into delivery planning by converting requirements into user stories, a prioritised backlog, and a sprint plan. The main challenge was prioritisation. It was necessary to distinguish between the minimum viable product and features that should be deferred. This discipline improved realism and prevented the backlog from becoming overly ambitious.

The GitHub Project template selection provided the practical delivery structure for the project. Selecting an appropriate template required balancing simplicity, automation, and traceability. The **Automated Kanban** template was the most suitable option because it supported sprint-based delivery, issue traceability, and manageable workflow progression without introducing unnecessary administrative overhead. The customised Kanban board, with additional **Testing** and **Blocked** columns, made work progress visible and created a clear link between user stories, issues, labels, and board columns.

## Project Management Reflection: Kanban, Jira, Trello
The board-management exercise also clarified the relative strengths of common project-tracking tools. **Kanban**, as implemented in GitHub Projects, was the most appropriate approach for this project because it offered a lightweight visual flow, direct integration with repository issues, and simple traceability between requirements and tasks. Its primary advantage was the ability to show work in progress clearly while keeping the workflow easy to maintain.

By comparison, **Trello** provides a similarly visual experience, but it is better suited to simple task tracking than to issue-linked software delivery. It is effective for basic organisation, although it offers less native alignment with code repositories, sprint artefacts, and formal traceability. **Jira**, by contrast, provides stronger workflow control, reporting depth, and enterprise-level configuration. However, that additional capability introduces complexity that is disproportionate to a small academic project.

Overall, GitHub Projects offered a balanced middle ground because it integrated naturally with repository issues, labels, and assignments. This demonstrated that delivery management is not separate from requirements engineering; it is part of the same evidence chain.

## Object State and Activity Workflow Reflection
The state and activity modelling work required a careful balance between precision and readability. If the diagrams contained too much detail, they became difficult to interpret; if they were too abstract, they failed to explain meaningful behaviour. The most useful approach was to model the lifecycles that matter most to the application, such as meal entries, nutrition goals, summaries, exports, and deployment configuration, while keeping the transitions aligned with the documented requirements.

The SVG exports made the diagrams easier to review in the repository itself and improved presentation consistency across the document set. The state diagrams clarified how data objects evolve over time, whereas the activity diagrams clarified how users and the system cooperate during a task. This distinction was important because the two diagram types answer different questions: state diagrams explain what can happen to an object, while activity diagrams explain how a process unfolds from start to finish.

The main challenge was ensuring that each diagram remained faithful to the current project scope. It was tempting to add additional states and branches for advanced functionality, but that would have weakened traceability to the existing requirements and made the documentation less coherent. Restricting the diagrams to the approved feature set produced a more formal and defensible result.

## Architecture Reflection
`ARCHITECTURE.md` provided the structural view of the application and connected user-facing requirements to implementation concerns. Although the document is conceptual, it adds value by explaining how the interface, data flow, and persistence layers interact. The architectural view supports implementation planning and improves maintainability because it makes system boundaries more explicit.

## Domain Modeling and Class Diagram Reflection
Designing the domain model required a more disciplined level of abstraction than the earlier requirements and use case documents. The main challenge was deciding which concepts should be treated as stable domain entities and which concepts should remain operational details. For example, a food item, a meal log, a nutrition goal, and a nutrition summary clearly represent enduring business concepts, whereas calculation routines and duplicate checks are better expressed as responsibilities or methods. Preserving that separation was important because a domain model becomes less useful when it mixes business concepts with implementation noise.

Another difficulty was determining the correct relationships between entities. The most important design choice was to treat `MealLog` as the central aggregate because meal entries are only meaningful in the context of a log. This made the composition relationship between `MealLog` and `MealEntry` a natural fit. By contrast, `FoodItem` was modelled as a referenced object rather than something owned by a meal entry, because the same food definition may be reused across many logs. That distinction improved clarity and avoided a design that implied unnecessary duplication of food metadata.

The addition of manually created food items also required careful treatment. Since the system now supports the option to add a food item to the database, the catalogue had to be represented as a first-class domain object rather than as an incidental storage detail. This decision helped align the structural model with the functional requirements and with the use case that permits users to extend the food catalogue when no suitable match exists. It also improved traceability, because the class diagram now shows how the catalogue supports both search and creation workflows.

Method definition presented a further trade-off. The diagram needed enough behavioural detail to be meaningful, but not so much detail that it became a pseudo-code document. I therefore included only methods that reflect core responsibilities, such as searching the catalogue, adding a food item, recalculating meal totals, and deriving summaries. This keeps the diagram readable while still communicating how the objects participate in the system. In particular, derived outputs such as `NutritionSummary` were deliberately modelled as computed structures rather than manually maintained records. That choice reduces the risk of inconsistency and aligns with the earlier requirements for accurate totals and summaries.

The class diagram also aligns well with the behavioural models produced earlier. State diagrams describe how objects change over time, whereas the class diagram explains what the objects are and how they relate structurally. For instance, the state of a meal log or nutrition goal may change, but the class diagram provides the static framework within which those transitions occur. Likewise, the activity diagrams show the flow of work during search, meal creation, and food-item creation, while the class diagram clarifies which objects are responsible for supporting those flows. This alignment strengthens traceability because the same concepts appear consistently across the requirements, use cases, state models, activity workflows, and structural design.

One useful lesson from this stage was that object-oriented design benefits from restraint. It is tempting to create a class for every possible action, report, or exception, but that approach usually leads to fragmentation. A more effective structure is to preserve a small set of cohesive classes with clear responsibilities and explicit relationships. In this project, that meant keeping the model centred on users, food items, meal logs, goals, and summaries rather than introducing unnecessary layers of inheritance. Composition and aggregation were therefore more appropriate than deep inheritance hierarchies, because the domain is better described through ownership and reuse than through subtype complexity.

Overall, the domain model and class diagram reinforced the importance of consistency across documentation. Each layer now tells the same story from a different perspective: stakeholders express the need, requirements define the behaviour, use cases show interaction, behavioural models show change and flow, and the class diagram shows the structural foundation. That coherence is the primary value of this stage of the work.

## README Reflection
`README.md` serves as the navigation layer for the repository. Its value lies in accessibility rather than depth. By consolidating the project documents in one place, it presents the repository as a coherent documentation set rather than a collection of unrelated files. For assessment purposes, this improves readability and demonstrates that the project has been organised intentionally.

## Conclusion
Taken together, the repository documents form a traceable progression from problem definition to verification and delivery planning. The main challenge throughout the work was maintaining consistency while adapting each document for a different purpose. The final result is stronger because each file supports the others. The documentation now presents a clear and formal account of the project’s scope, requirements, use cases, testing strategy, and Agile execution approach.
